# 토스애즈 의사결정 엔진 (Decision Engine)

이 문서는 토스애즈 리드 최적화 운영체계의 핵심인 **의사결정 엔진(Decision Engine)**의 구조와 동작 방식을 설명합니다.

## 1. 아키텍처 개요
의사결정 엔진은 Python 워커(`python_worker/decision_engine.py`) 내에 구현되어 있으며, Node.js 백엔드와 MSA 형태로 결합되어 동작합니다.

1. **지표 수집 (Node.js)**: 15분 주기 스케줄러가 활성화된 Arm의 성과 지표(노출, 클릭, 리드, CPA 등)를 수집하여 `EVALUATE_ARMS` 작업을 생성합니다.
2. **평가 (Python Worker)**: Python 워커가 작업을 수신하면 `DecisionEngine`을 통해 각 Arm의 상태와 추천 액션을 계산합니다.
3. **결과 반영 (Node.js)**: 평가 결과가 반환되면 Node.js는 이를 `RecommendedAction` 테이블에 저장합니다. 자동 실행 가능한 액션은 즉시 승인(`APPROVED`) 처리되고, 수동 검토가 필요한 액션은 대기(`PENDING`) 상태로 남습니다.

## 2. 평가 로직 (Evaluation Logic)
엔진은 다음 세 가지 주요 로직을 순차적으로 적용합니다.

### 2.1 샘플 부족 보호 (Sample Protection)
데이터가 충분히 쌓이기 전에는 성급한 판단을 유보합니다.
* **조건**: 소진 비용이 목표 CPA의 50% 미만이거나, 클릭 수가 40회 미만인 경우.
* **결과**: `TESTING` 상태 유지, 액션 보류(`HOLD`).

### 2.2 하드 룰 (Kill Switches)
비정상적인 비용 소진을 막기 위한 즉각적인 방어 기제입니다.
* **조건 1**: 클릭 40회 이상 발생했으나 리드 0건.
* **조건 2**: 소진 비용이 목표 CPA의 1.5배를 초과했으나 리드 0건.
* **결과**: `PAUSE_CANDIDATE` 상태로 전환, 중단(`PAUSE`) 액션 추천. (단, **수동 승인** 필요)

### 2.3 베이지안 확률 및 추세 평가 (Bayesian & Trend Analysis)
단순 평균의 함정을 피하기 위해 `scipy.stats.beta` 분포를 활용하여 성공 확률을 계산하고, 최근 1일/3일 CPA 추세를 비교합니다.
* **성공 확률 계산**: 현재까지의 클릭/리드 데이터를 바탕으로, 이 Arm의 실제 CVR이 목표 CVR보다 높을 확률을 계산합니다.
* **증액 (SCALE)**: 성공 확률 > 80% 이고, 3일 평균 CPA가 목표 이하인 경우 예산 15% 증액. (자동 실행)
* **감액 (REDUCE)**: 성공 확률 < 30% 이거나, 3일 평균 CPA가 목표의 1.2배를 초과하는 경우 예산 20% 감액. (자동 실행)
* **수동 검토 (MANUAL_REVIEW)**: 3일 평균 CPA는 좋으나, 최근 1일 CPA가 30% 이상 급등한 경우 소재 피로도로 간주하여 수동 검토 요청.

## 3. 예산 배분 (Explore & Exploit)
* **Exploit (80%)**: `SCALE`, `STABLE` 상태의 검증된 Arm에 예산을 집중합니다.
* **Explore (20%)**: `TESTING` 상태의 신규 Arm에 최소한의 트래픽을 보장하여 데이터를 수집합니다.

## 4. 자연어 설명 생성
모든 추천 액션은 관리자가 쉽게 이해할 수 있도록 자연어 사유(`explanation`)를 함께 생성합니다.
* 예: *"성공 확률이 92.4%로 매우 높고 3일 CPA(9000원)가 목표치 이하입니다. 최근 1일 추세도 긍정적이므로 예산을 15% 증액합니다."*
